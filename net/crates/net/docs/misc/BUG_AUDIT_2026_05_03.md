## Critical

1. **[FIXED — durable_rename helper]** **`adapter/net/redex/disk.rs:1322` — `compact_to` is non-atomic on Windows.** `fsync_dir` is a no-op on `cfg(not(unix))`. Combined with the cross-file rename sequence (idx + dat + ts), a power loss can leave the directory pointing at mismatched inodes. Fix: introduced `durable_rename` helper. On Windows, routes through `MoveFileExW` with `MOVEFILE_REPLACE_EXISTING | MOVEFILE_WRITE_THROUGH` so each rename blocks until the NTFS journal commits — closing the per-call durability hole that the stdlib `fs::rename` leaves open. The cross-file mixed-state window between two of the three renames is unchanged on either platform; that remains the deferred manifest-pointer rework.

2. **[FIXED in `f18dc900`]** **`bus.rs:686-711` (`remove_shard_internal`) — Batch worker awaited before drain worker, contradicting the function's own doc-comment.** Lines 506-520 explicitly document the correct teardown order ("drain first, so the in-progress `scratch` and `sender.send(batch).await` are flushed before the batch worker exits"); `add_shard_internal` rollback (533-569) and `shutdown_via_ref` (1329-1349) follow that order. The actual code awaits batch first, then drain. Drain worker still owns a sender clone, so `recv()` doesn't return `None` until drain exits — and there is no `tokio::time::timeout(...)` shell here (unlike the rollback path), so a wedged adapter on the shard being removed pins this function indefinitely. Fix: swap the await order to drain → batch, or wrap in the same timeout the rollback path uses.

3. **[FIXED in `781620b5`]** **`ffi/mod.rs:1322-1331` (`net_poll_ex`) — Panic across FFI on OOM.** `event.id.as_bytes().to_vec().into_boxed_slice()` and `event.raw.to_vec().into_boxed_slice()` can panic mid-loop, leaking earlier `Box::into_raw`s + the `std::alloc::alloc(layout)` array. No `catch_unwind`. Fix: wrap entry points in `catch_unwind`, or use `try_reserve`.

4. **[FIXED in `3efa4a9d`]** **`ffi/mod.rs:805-841` (`net_poll`) — Buffer-size check runs after events have been consumed.** `bus.poll(request)` advances the consumer cursor *before* the response is serialized; if `response_json.len() + 1 > buffer_len`, the function returns `BufferTooSmall` and drops the entire `response`. The caller's contractual retry fetches starting from the now-advanced cursor — every event in the failed serialization is silently lost. The C-API doc itself shows fixed-size buffers, so the trigger is the recommended usage pattern. Fix: serialize first and size-check before advancing the cursor, or buffer the response so a retry can resume.

## High

5. **[FIXED in `41be4ff2`]** **`adapter/net/redex/disk.rs:1356-1413` — Temp-file leak on `compact_to` failure.** Cleanup at 1411-1413 runs *after* `open_or_poison(...)?` (1356-1358) and `clone_or_poison(...)?` (1396-1398). On any post-rename reopen failure the three placeholder files in `/tmp` leak forever.

6. **[FIXED — applied/folded watermark split]** **`adapter/net/cortex/adapter.rs:486-499` — Watermark advances past unfolded events under `Stop` policy.** `recoverable_decode` causes `folded_through_seq.store(seq)` for events whose state mutation never landed; `wait_for_seq(seq)` returns true incorrectly. Combined with snapshot-restore (#7) this produces permanent state divergence. Fix: introduced `applied_through_seq` as a strict-prefix watermark — advances only on `Ok(())` AND only when this seq is the immediate successor of the current value. Any skip (recoverable_decode under Stop, or any error under LogAndContinue) caps it at the prior consecutive seq forever. `folded_through_seq` keeps current semantics so live consumers (DoS-resistance contract) don't deadlock; `wait_for_seq` doc tightened to surface the truth.

7. **[FIXED — same change as #6]** **`adapter/net/cortex/adapter.rs:374-386` + `tasks/adapter.rs:340-344` — Snapshot persists `last_seq` for skipped events.** When (#6) fires, `snapshot()` returns `last_seq = Some(N)` for events whose state was never applied; restore tails from `N+1` and the skipped seq is never re-attempted. Violates "log is the source of truth." Fix: `snapshot()` now returns `applied_through_seq` (strict-prefix); restore tails from there and re-attempts every seq past the gap. Trade-off documented on `snapshot`: non-idempotent folds will double-apply intervening successful seqs, which is a strictly-better failure mode than the pre-fix permanent loss.

8. **[FIXED — compute_checksum_with_meta]** **`adapter/net/cortex/meta.rs:135-137` — Checksum doesn't cover meta header.** `compute_checksum` hashes only `tail`. A bit-flip in `dispatch` byte (e.g. `STORED` → `DELETED`) is undetected and silently re-routes the event to the wrong fold arm. Fix: introduced `compute_checksum_with_meta(meta, tail)` that hashes the 20-byte header (with the `checksum` slot zeroed) followed by the tail. Memories + tasks producers stamp v2 checksums on every new write; corresponding fold consumers verify v2 first and fall back to legacy v1 (tail-only) for records written by pre-fix adapters. New writes detect dispatch / flags / origin_hash / seq_or_ts bit-flips; old records keep working with the original tail-only-coverage limitation. Migration is one-way (downgrade to a pre-fix adapter binary will skip every event written by a v2-capable producer because the legacy verifier expects checksum == xxh3(tail)).

13. **[FIXED — DaemonError::Stale + per-mutator guard_identity]** **`adapter/net/compute/registry.rs:71-74` (`replace`) — Not safe for stateful in-flight work.** A caller holding an old `Arc<Mutex<DaemonHost>>` from `get_arc` keeps mutating the old host while new traffic hits the new one; old chain's writes are dropped. `replica_group::on_node_failure`, `fork_group::on_node_failure`, and `standby_group::on_node_failure` all use this pattern without quiescing. Fix: accept an `out_failed_indices: *mut size_t, out_failed_len: *mut size_t` pair to surface the exact dropped indices.

105. **[FIXED in `928bd520`]** **`ffi/mod.rs:180-186, 1048-1053` — `bus_taken` defense-in-depth claim is doc-only.** The doc on `bus_taken` says "FFI ops also check this before touching `bus`/`runtime`," but a grep finds the field read only inside `net_shutdown`. 106. **[FIXED in `64dd771c`]** **`bus.rs:621-624 / 779-782` (`poll_merger` swap) — Lost-update race between concurrent add/remove.** Two callers (e.g. scaling-monitor `add_shard_internal` racing `manual_scale_down → remove_shard_internal`) read `shard_manager.shard_ids()` and then race on `self.poll_merger.store(...)`. T1 reads `{0..5}`, T2 reads `{1..4}`, T2 stores `{1..4}`, T1 stores `{0..5}` — final published view includes a removed shard 0. The merger then routes polls to a removed shard until the next topology change repairs the snapshot. Fix: serialize the `shard_ids() → store` block under a mutex.

107. **[FIXED in `e42dcc15`]** **`bus.rs:1245-1271` (lossy-shutdown bookkeeping) — `events_dropped` incremented for events that may still be delivered.** On the in-flight deadline path, `stranded = self.in_flight_ingests.load(SeqCst)` is added to `events_dropped`. But many of those producers will still successfully complete `shard_manager.ingest()` (their event lands in the ring buffer) AND be drained on the final two-pass sweep. Those events get counted in both `events_ingested` and `events_dropped`, violating the invariant `events_ingested = events_dispatched + events_dropped` and making `shutdown_was_lossy` a false positive on every deadline-triggered shutdown. Fix: subtract the events the final sweep actually drained, or replace the heuristic with an explicit "events still in ring at deadline" count.

108. **[FIXED in `10f61560`]** **`shard/mapper.rs:172-177` (`record_flush`) + `246-247` (`collect_and_reset`) — Latency-sum/count desync zeroes the average that drives scale-up.** `record_flush` does two independent `fetch_add`s; `collect_and_reset` does two independent `swap`s. If a metrics tick runs between the two `fetch_add`s, the swap of `flush_latency_sum_us` captures a sum that includes a flush whose count is captured in the *next* window. The doc-comment acknowledges the inaccuracy as "an accepted trade-off," but the real impact is that `avg_flush_latency = sum.checked_div(count).unwrap_or(0)` returns 0 in window N (sum without count) and 0 in window N+1 — the average that `evaluate_scaling`'s flush-latency trigger reads silently zeros out under sustained load, suppressing the scale-up decision. Same race for `push_latency_sum_ns` / `push_count`. Fix: pack `(sum, count)` into a single `u128` and CAS together.

109. **[FIXED in `288571ef`]** **`ffi/mod.rs:372-380` (`parse_config_json`) — Unknown `backpressure_mode` silently falls back to `DropNewest`.** `parse_poll_request_json` was specifically fixed (with a regression test) to surface unknown / wrong-typed enum strings as `InvalidJson`; this parallel parser silently swallows typos and selects `DropNewest`. A caller writing `"DropOldset"` (typo) or `"FailProduce"` gets a different durability profile with no error at deploy time. Same parser also has no support for `BackpressureMode::Sample { rate }`. Fix: error on unknown values, and add a `Sample` arm with `rate` validation.

110. **[FIXED in `426d325b`]** **`ffi/cortex.rs:225-239` (`net_redex_open_file`) — `retention_max_*` accept zero, fsync params do not.** Three lines earlier the parser explicitly rejects `(Some(0), _)` / `(_, Some(0))` for fsync; `retention_max_events`, `retention_max_bytes`, and `retention_max_age_ms` are propagated unchecked. Setting `retention_max_events = 0` means "evict everything immediately on first append" — almost certainly a config mistake intended as "no limit," especially since most JSON config schemas use `null` or omission for "unlimited." Inconsistent and silent data loss. Fix: reject `Some(0)` for the retention triple at the same gate that rejects fsync zeros.

111. **[FIXED in `811e4ff3`]** **`ffi/mod.rs:490-498` and `ffi/mesh.rs:411-417` — Net `heartbeat_interval_ms` / `session_timeout_ms` and mesh `heartbeat_ms` accept `0`.** `EventBusConfig::validate` is careful to reject zero `Duration`s for `cooldown`, `metrics_window`, `scale_down_delay`, `adapter_timeout`, etc. The Net adapter and mesh JSON parsers have no equivalent zero-check at the FFI boundary — a `0` flows through to `Duration::from_millis(0)`.
